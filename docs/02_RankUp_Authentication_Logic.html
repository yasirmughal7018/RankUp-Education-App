1. Executive Summary

  • Authenticated API calls use a Bearer JWT access token (~30 minutes). Refresh tokens last 30 days and are stored hashed.
  • Clients use a two-step login: POST /api/auth/login-status (identifier) → then either set-initial-password or password login.
  • Username priority: CNIC if provided, otherwise mobile. Login lookup order: username → cnic → mobile_number.
  • Self-service registration creates an inactive app_users row + role in app_user_roles + pending rows in app_user_approval. Only PortalAdmin activates. User then sets password via set-initial-password (admin never sets the password on approve).
  • Reject is a soft reject (rejected_at); unique indexes ignore rejected rows so the person can re-request.
  • Roles live in app_user_roles (multi-role where allowed). Session role is on the JWT and refresh_tokens.active_role. Student and PortalAdmin are exclusive (cannot combine with any other role). SchoolAdmin, CampusAdmin, Teacher, and Parent may combine.
  • School/campus change (Teacher / Student / CampusAdmin) locks the account until an allowed admin applies or rejects the request. Parent cannot request school/campus change.
  • admin_target and app_users.role columns are dropped / unused.

2. Username & Login Rules

2.1 Username priority

  • If CNIC is provided → username = CNIC (and cnic is set).
  • If CNIC is absent → username = mobile_number.
  • On PortalAdmin activation, if CNIC is already on the request and username differs, username is updated to CNIC (when free among non-rejected rows).

2.2 Login identifiers

GetByLoginIdentifierAsync(identifier):
  1. Match app_users.username
  2. Else match app_users.cnic
  3. Else match app_users.mobile_number
  Prefer non-rejected rows (PreferActiveRegistrationAsync), then newest Id.
  Clients send the identifier in the username JSON field.

2.3 Uniqueness (soft-reject aware)

Check / indexRule
UsernameExistsAsync / ix_app_users_username_activeUnique among rows where rejected_at IS NULL
CnicExistsAsync / ix_app_users_cnic_activeUnique among non-rejected rows when CNIC set
MobileNumberExistsAsyncExists check skips rejected rows

2.4 Parent vs Student / Teacher on registration

RoleSchool / campus / rollApproval queue at submit
ParentNo school, campus, or rollPortalAdmin only
Student / Teacher — no schoolOptional school/campus omittedPortalAdmin only
Student / Teacher — school onlySchool required for school pathSchoolAdmin (that school) + PortalAdmin
Student / Teacher — school + campusCampus selectedCampusAdmin (campus) + SchoolAdmin (school) + PortalAdmin

Routing is derived at submit time into app_user_approval — not via admin_target.

2.5 must_change_password

ValueMeaning
null or falseNo forced password change
trueSet on PortalAdmin approve / directory provision; signals password setup needed
After set-initial-password or change-passwordCleared to false
After first password via set-initial-password, the user is Ready to sign in — there is no additional forced change-password modal for that path. The React force-change modal applies when an existing session has mustChangePassword === true (e.g. admin-driven reset flows).

3. Account States & login-status

StateDB conditionlogin-statusCan sign in?
Pending registration is_active=false AND password_hash NULL AND rejected_at NULL PendingApproval No
Rejected (soft) Row kept; rejected_at set Rejected No — may submit a new registration
Approved, needs password is_active=true AND password_hash NULL NeedsPasswordSetup No — call set-initial-password first
Ready is_active=true AND password set Ready Yes — password login
Locked (school/campus change) is_active=false, password set, pending app_user_school_change_request LockedPendingSchoolChange No — locked page / message
Deactivated is_active=false, password set, no pending school change Throws “This account is not active.” (not a status enum) No 
Domain helpers (User.cs):
  IsPendingRegistration = !IsActive && no password && !IsRejectedRegistration
  NeedsPasswordSetup    = IsActive && no password
  IsRejectedRegistration = RejectedAt != null
While SchoolAdmin / CampusAdmin have recorded registration approval but PortalAdmin has not, login-status remains PendingApproval. Soft-approve does not activate the account.

4. Login, Set Initial Password, Switch Role

4.1 Login flow (clients)

1. User enters CNIC or mobile
2. POST /api/auth/login-status { username }
3. Branch on Status:
     PendingApproval           → show wait message (stay on identifier)
     Rejected                  → show reject message; offer Request Access
     NeedsPasswordSetup        → show set-password step
     LockedPendingSchoolChange → locked UI (React: /account-locked; Mobile: in-login locked step)
     Ready                     → show password step
4a. NeedsPasswordSetup → POST /api/auth/set-initial-password
       → then password login
4b. Ready → POST /api/auth/login { username, password }
       → JWT + refresh + CurrentUser (active role)
       → RecordLogin / last_login_at updated

4.2 Password login gates (API)

  1. GetByLoginIdentifierAsync
  2. EnsureCanLogin() — blocks deleted, rejected, pending, inactive
  3. If NeedsPasswordSetup → error: set password on login screen first
  4. Verify password (PBKDF2)
  5. Issue refresh token for active role + access JWT; update last_login_at

4.3 Set initial password

  • POST /api/auth/set-initial-password — anonymous; requires identifier + new password (min 6 chars).
  • User must be active and NeedsPasswordSetup.
  • Sets password_hash and clears must_change_password (false).

4.4 Multi-role / switch-role

Rules are enforced in UserRoleRules (Domain) when adding a role to an account.

RuleDetail
ExclusiveStudent — cannot combine with any other role
ExclusivePortalAdmin — cannot combine with any other role
May combineSchoolAdmin, CampusAdmin, Teacher, Parent (any mix of these)

Examples allowed: Teacher+Parent; CampusAdmin+Teacher; SchoolAdmin+CampusAdmin+Teacher/Parent.

  • Roles are stored in app_user_roles only.
  • Default login active role = earliest assignment in app_user_roles (by created_at).
  • POST /api/auth/switch-role (JWT) issues a new access + refresh token scoped to the selected role.
  • Refresh tokens store active_role; refresh reuses that role for the new access token.
  • Authorization uses the active role only.
  • UI: React role switcher when roles.length > 1; Mobile Profile role switcher when multi-role.

5. Logout & Token Revoke

  • POST /api/auth/logout — optional body with refresh token. When provided, that token is revoked (hash lookup). Missing token is a no-op success.
  • React AuthProvider.logout and Mobile ApiAuthRepository.logout send the refresh token then clear local session.
  • Also revoke-all for the user on: school/campus change request (lock) and account deactivate — undocumented as “logout” but required for security.

5.1 Admin-mediated password reset

  1. User → POST /api/auth/password-reset/request (CNIC/mobile). Always returns success (no existence leak).
  2. If an eligible active account with a password exists, admins who can Clear that user get PasswordResetRequest (school/campus candidates filtered by Clear rules — not identical to registration reviewers).
  3. Admin → Clear password from the notification bell (or POST /api/auth/password-reset/clear) → password cleared, sessions revoked, NeedsPasswordSetup.
  4. User → login-status → set-initial-password → sign in.
TargetNotified / can Clear
PortalAdminPortalAdmin only
SchoolAdminPortalAdmin + SchoolAdmin of that school
CampusAdminPortalAdmin + SchoolAdmin of that school (not peer CampusAdmins)
Teacher / StudentPortalAdmin + SchoolAdmin + CampusAdmin of destination scope
Parent (no school)PortalAdmin only

CampusAdmin cannot clear SchoolAdmin or CampusAdmin. Only PortalAdmin can clear PortalAdmin. Clear UI is Web-only.

6. Registration & Pending Detection

6.1 Self-service register

Client → POST /api/auth/register
  → username = CNIC ?? mobile
  → uniqueness among rejected_at IS NULL
  → INSERT app_users (is_active=false, password_hash=null)
  → INSERT app_user_roles (requested role)
  → INSERT app_user_approval rows for eligible reviewers
  → notify eligible admins (app_notifications, RegistrationRequest)

6.2 Pending detection

  • Domain: User.IsPendingRegistration (inactive + no password + not rejected).
  • Admin list: GET /api/auth/registrations/pending — PortalAdmin / SchoolAdmin / CampusAdmin; scoped by school/campus where applicable; response includes pendingApprovers and currentUserHasApproved.
  • Login: login-status → PendingApproval.
  • Directory-created users are active immediately with no password (NeedsPasswordSetup) and do not use app_user_approval.

7. Approval Queue & Soft Reject

7.1 Who can act

ActionPortalAdminSchoolAdminCampusAdmin
List pending registrationsYesYes (scoped)Yes (scoped)
Record approve (soft)—YesYes
Activate accountYes onlyNoNo
Soft-rejectYesYes (scoped)Yes (scoped)

7.2 Soft-approve vs PortalAdmin activate

  • SchoolAdmin / CampusAdmin: mark their app_user_approval row only. Account stays pending; login-status remains PendingApproval until PortalAdmin activates. UI shows “Approved — awaiting Portal Admin” when currentUserHasApproved.
  • PortalAdmin: marks their row; creates slim role profile (student defaults: grade 1, section A); calls ApprovePendingRegistration() → is_active=true, password still null, must_change_password=true. Other pending approval rows are not required.
  • Approve request body is empty — no admin password, no academic override payload.

7.3 “Pending with” filter

  • Pending list returns pendingApprovers from ListPendingApproversForUserAsync: queue rows where is_approved IS NULL.
  • If any CampusAdmin has already soft-approved (is_approved = true), pending SchoolAdmin rows are omitted from “Pending with” — CampusAdmin approval covers SchoolAdmin for display (SchoolAdmin is no longer required).
  • PortalAdmin is still required to activate. CampusAdmin soft-approve alone does not activate the account.

7.4 Soft reject

  • Sets rejected_at; keeps user row and approval trail.
  • Rejector’s app_user_approval marked rejected.
  • login-status → Rejected. User may submit a new registration (new row).

8. School / Campus Change Lock

  • Who may request: Teacher, Student, CampusAdmin. Cannot: PortalAdmin, SchoolAdmin, Parent.
  • Confirm on Profile → POST /api/auth/me/school-change → pending request → is_active=false → all refresh tokens revoked → client logout → locked messaging.
  • login-status → LockedPendingSchoolChange.
  • Request UI: React /account and Flutter Mobile Profile (Teacher / Student / CampusAdmin).
  • Admin apply / reject queue: Web only — /admin/directory/school-changes. Mobile does not implement the admin school-change queue.

8.1 Apply matrix

ApproverCan apply (unlock + move)
PortalAdminAny pending request
SchoolAdminTeacher / Student / CampusAdmin into destination school
CampusAdminTeacher / Student into own campus (not CampusAdmin→CampusAdmin peers)

List filters match this matrix: every admin who can see a pending request can Approve & apply (or Reject). Reject unlocks without changing school/campus. Admin UI (Web): /admin/directory/school-changes.

11. JWT, Refresh Tokens, Security

11.1 JWT claims

ClaimSource
sub / userIdapp_users.id
nameapp_users.username
role (also ClaimTypes.Role)Active session role (enum name)
rolesAll assigned roles, comma-separated (from app_user_roles)
permissionsAuthPermissions.ForRole(activeRole), comma-separated
profileIdRole profile id when loaded
schoolId / campusIdUser school/campus

role = active session role; roles = full assignment list. Authorization uses the active role. mustChangePassword is returned on login / me / change-password responses — not as a JWT claim.

11.2 Security

  • Passwords: PBKDF2 via PasswordHasher.
  • Refresh tokens stored as SHA-256 hex only; lifetime 30 days.
  • Access token: Jwt:AccessTokenMinutes (default 30).
  • Login rate limit: 8 requests per 60 seconds.
  • Password-reset request does not reveal whether the identifier exists.
