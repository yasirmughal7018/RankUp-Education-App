1. Executive Summary

  • All authenticated API calls use a Bearer JWT access token.
  • Login accepts CNIC or mobile number (also matches stored username). Only users with is_active = true and a non-null password_hash may sign in.
  • Username priority: CNIC if provided, otherwise mobile number. When CNIC is set (at register or approve), username becomes the CNIC.
  • Registration creates an inactive app_users row with identity fields persisted. Admin approval activates the account, sets a password, forces password change, and creates the role profile row.
  • admin_target: School Admin → School Admin + Portal Admin (SuperAdmin) can approve; Portal Admin → Portal Admin only. Eligible admins receive an in-app notification.
  • must_change_password: true = must change; null/false = no force. After the user changes password once → false.
  • Five roles: SuperAdmin (Portal Admin), SchoolAdmin, Teacher, Student, Parent.
  • Clients use one shared login screen. Role comes from the API response — not selected on the login form.
Current model: Registration fields (mobile, CNIC, email, school/campus, roll/teacher code, reason, admin target) are persisted on app_users. School/campus/CNIC/roll/teacher code are owned by app_users; profile tables hold role-specific academic fields only (e.g. student grade/section).

4. Username & Login Rules

4.1 Username priority

  1. If CNIC is provided → username = CNIC (and cnic column is set).
  2. If CNIC is absent → username = mobile_number.
  3. On approval, if CNIC is set/updated → username is updated to CNIC when needed.

4.2 Login identifiers

Clients send the identifier in the username JSON field. Server resolves via GetByLoginIdentifierAsync in this order:

  1. Exact match on app_users.username
  2. Else match on app_users.cnic
  3. Else match on app_users.mobile_number

Users can therefore sign in with CNIC or mobile number.

4.3 must_change_password

ValueMeaning
null or falseNo forced password change
trueUser must change password (set on admin approval)

After the user successfully calls POST /api/auth/change-password once → field is set to false. React portal blocks the UI with a change-password modal when the value is strictly true.

Client (React Request Access / Mobile sheet) → POST /api/auth/register → AuthService.RegisterAccountAsync() 1. username = CNIC ?? mobile 2. admin_target = schoolId ? "School Admin" : "Portal Admin" 3. INSERT app_users (pending: is_active=false, password_hash=NULL) 4. ListAdminRecipientsAsync(admin_target, school_id) 5. Create in-app notifications (RegistrationRequest) → RegisterAccountResponse { id, username, fullName, role }

9. Login Request Flow

Client LoginPage → POST /api/auth/login { username: <CNIC or mobile>, password } → AuthService.LoginAsync() 1. UserRepository.GetByLoginIdentifierAsync() — username → cnic → mobile 2. Attach profile context (profileId; school/campus from app_users) 3. user.EnsureCanLogin() — is_active + password_hash required 4. PasswordHasher.Verify() 5. IssueRefreshToken() → INSERT refresh_tokens (hashed) 6. JwtTokenService.CreateAccessToken() → ApiResponse<LoginResponse> including mustChangePassword → Client saves tokens + user; React may show Change Password modal

10. Token Refresh & Logout

10.1 Refresh

Client POST /api/auth/token/refresh { refreshToken } → Hash token (SHA-256 hex) → Lookup refresh_tokens by token_hash → Verify not expired/revoked → Revoke old token → Issue new refresh + access JWT → Return AuthTokensResponse

10.2 Logout

Client POST /api/auth/logout { refreshToken } → Revoke matching refresh_tokens row → Client clears token store + local user cache

10.3 Access token lifetime

Configured in appsettings.json → Jwt:AccessTokenMinutes (default 30 minutes). Refresh token lifetime is 30 days in AuthService.

14. Security & Rate Limiting

  • Passwords hashed with PBKDF2-SHA256 (100,000 iterations) via PasswordHasher.
  • Refresh tokens stored as SHA-256 hex only — raw token returned once to client.
  • Login endpoint rate-limited: LoginPermitLimit = 8 per 60 seconds.
  • Password reset always returns success message to prevent identifier enumeration.
  • JWT signing key must be ≥ 32 characters — configure in Jwt:SigningKey.
  • CORS policy RankUpEducation applied for configured origins.
