1. Target business rules

Admin Portal in product language = role PortalAdmin (lookup id 2010).

  1. 1. PortalAdmin full control: PortalAdmin (Admin Portal) can Create, Update, Delete any question; Activate / Deactivate / Archive; Approve; and Reject (rejection reason required).
  2. 2. Create: SchoolAdmin, CampusAdmin, Teacher, and Parent can create questions.
  3. 3. Update own (pre-approval): SchoolAdmin, CampusAdmin, Teacher, and Parent can update their own questions only while status is not Approved (Draft, PendingReview, Rejected).
  4. 4. Delete own (pre-approval): SchoolAdmin, CampusAdmin, Teacher, and Parent can delete their own questions only while status is Draft, PendingReview, or Rejected (not Approved).
  5. 5. Delete after Approved: Once Approved, only PortalAdmin may delete. Non–PortalAdmin cannot delete Approved questions (even their own).
  6. 6. Activate / Deactivate / Archive: Only PortalAdmin can Activate, Deactivate, or Archive questions. No other role.
  7. 7. View Approved (creators + admins): PortalAdmin, SchoolAdmin, CampusAdmin, Teacher, and Parent can view all Approved questions with options/answers and full details. Students cannot open the question bank at all — they only see questions inside quizzes.
  8. 8. Approve / Reject = PortalAdmin only: Only PortalAdmin approves or rejects. Reject requires a non-empty rejection reason. After Reject: owner may edit and resubmit to PendingReview, or delete the question.
  9. 9. Excel import (Web only): Import from Excel is available on Web for create-capable roles (PortalAdmin, SchoolAdmin, CampusAdmin, Teacher, Parent). Not available on the Mobile app.

Rule numbers match the product request (item 6 was unused in the source list).

7. Open decisions before implement

  1. Import default status: Draft (implemented) — owner submits to PendingReview.
  2. Delete while linked to quizzes: blocked for everyone including PortalAdmin (implemented).
  3. Rejected edit stays Rejected until Submit for review (implemented).
  4. IsAiApproved dual gate dropped — PortalAdmin Approve sets quiz-eligible (implemented).

8. Suggested improvements

Advisory recommendations to strengthen question logic (not all required for v1):

  • Drop IsAiApproved dual gate: One PortalAdmin Approve should make a question quiz-eligible. The current human + AI two-step adds friction without a real LLM; keep optional quality checks as soft warnings on Approve instead.
  • Canonical status IDs only: Stop matching loose aliases (Active/Published/Under Review). Use 110–114 only so QA, UI badges, and DB stay aligned.
  • Reject reason UX: Require min length (e.g. 10 chars), show reason on owner’s detail card, and clear reason when resubmitted to PendingReview.
  • Resubmit as explicit action: Prefer Edit (stays Rejected) + Submit for review → PendingReview, rather than auto-flipping status on every save. Avoids accidental re-queue.
  • Quiz-link delete policy: Block delete (even PortalAdmin) while quiz_questions links exist; offer “Remove from N quizzes” then delete. Prevents broken attempts mid-term.
  • Import validation report: Return row-level errors (row #, field, message); support dry-run; template .xlsx download on Web. Cap batch size (e.g. 200 rows).
  • Version or clone on Approved edit: If PortalAdmin edits an Approved question already used in quizzes, consider clone-as-new-version so historical attempts stay stable.
  • Pending queue filters: PortalAdmin queue: filter by class/subject/creator/date; bulk Approve/Reject with same reason for Reject.
  • Correct-answer visibility: Show correct answers in bank detail for manage roles; never expose correct answers to Student except after quiz submit/review rules already defined for attempts.
  • Audit trail: Log Approve/Reject/Activate/Archive with actor + timestamp (and reject reason) for disputes and QA.

9. QA scenarios

Q-01PortalAdmin CRUD + Approve + Activate

  1. PortalAdmin creates a question → Approve → status Approved.
  2. PortalAdmin Activate / Deactivate / Archive → allowed; other roles → forbidden.
  3. PortalAdmin updates/deletes any question → allowed (respect quiz-link policy if still enforced).
Expected: Full control for PortalAdmin including Activate/Deactivate/Archive.

Q-02Teacher creates and edits own Pending

  1. Teacher creates question → PendingReview (or Draft then submit).
  2. Teacher updates own Pending question → allowed.
  3. Teacher deletes own Pending question → allowed.
Expected: Owner can manage pre-Approved own items.

Q-03Teacher blocked on Approved

  1. PortalAdmin approves Teacher’s question.
  2. Teacher tries Update → forbidden.
  3. Teacher tries Delete → forbidden.
  4. Teacher tries Activate/Deactivate → forbidden.
Expected: Approved is locked for non–PortalAdmin; lifecycle flags are PortalAdmin-only.

Q-04CampusAdmin create

  1. Login as CampusAdmin → Create question succeeds.
  2. CampusAdmin sees own Pending; can edit/delete until Approved.
Expected: CampusAdmin is a first-class creator.

Q-05SchoolAdmin cannot approve or reject

  1. SchoolAdmin opens pending queue → no Approve/Reject (or API 403).
  2. PortalAdmin Approves → status Approved.
Expected: Approval/rejection is PortalAdmin-only.

Q-06Approved visible to creators; Student blocked from bank

  1. Create + Approve a question with options.
  2. As Teacher / Parent / SchoolAdmin / CampusAdmin: open Approved → see text, options, correct answers, explanation.
  3. As Student: no Questions bank menu; API list/get bank → forbidden. Student still answers questions inside a quiz attempt.
Expected: Bank read for manage roles; Students only via quizzes.

Q-07Excel import — Web only

  1. On Web, creator clicks Import → uploads valid .xlsx.
  2. Rows create questions owned by importer; status Draft or PendingReview (never Approved).
  3. Invalid rows reported with row errors.
  4. On Mobile: no Import button / no import API usage from app.
Expected: Bulk create from Excel on Web only.

Q-08Reject requires reason; owner resubmit or delete

  1. PortalAdmin Reject without reason → validation error.
  2. PortalAdmin Reject with reason → Rejected; reason stored and shown to owner.
  3. Owner edits Rejected question and submits → PendingReview.
  4. Alternatively owner deletes Rejected question → allowed.
Expected: Reject is reasoned; owner can fix/resubmit or remove.

Q-09Parent / SchoolAdmin ownership consistency

  1. Parent creates → edits own Pending → cannot edit after Approve.
  2. SchoolAdmin same; cannot approve/reject/activate others’ questions.
Expected: Ownership + PortalAdmin-only governances consistent.
