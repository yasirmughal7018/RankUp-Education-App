RankUp Education Mobile App

Login Screen Functionality Brief | Generated June 27, 2026

Executive Summary

  • One common login screen is used for Student, Parent, and Teacher users.
  • Users enter only Username or ID and Password.
  • The app does not show role buttons on the login form.
  • Role detection happens after authentication through the AppUser.role value.
  • No mobile registration and no OTP login/registration flow are implemented.
  • Password reset and account access are admin-request workflows.
  • Current development builds use mock repositories by default.

Single Login And Role Detection Flow

  1. User enters Username or ID and Password.
  2. LoginPage calls AuthController.login.
  3. The selected repository authenticates using mock data or the real API.
  4. The repository returns AuthSession with AppUser.role.
  5. AuthController stores the user in AuthState.
  6. GoRouter refreshes and redirects by role.

Password Reset Flow

  • Forgot password? opens a bottom sheet.
  • Username or ID is required.
  • Submitting calls requestPasswordReset(identifier).
  • Real API mode posts to /auth/password-reset/request.
  • Mock mode simulates the operation with a short delay.
  • Success message: Password reset request sent to admin.
  • No OTP verification is implemented.

Validation, Security, And Feedback

  • Login fields are editable and keyboard-enabled.
  • Password reset and account request required fields show Required when empty.
  • Auth operations use loading, success, and error states.
  • Password text is obscured.
  • Autocorrect and suggestions are disabled for login fields.
  • Unauthenticated protected routes redirect to /login.
  • Tokens are stored through TokenStore in API mode.

Anomalies And Gaps Found

  • Mock restoreSession returns a Student session if a token exists but the in-memory mock session is gone.
  • Email and mobile fields do not yet enforce format validation.
  • Password reset and account access are request-only flows; they do not complete account creation or password change in the app.
  • Unknown role values currently parse as student; backend role validation should be strict.
  • SystemChannels.textInput.show is used as an emulator keyboard workaround and should be monitored on real devices.

Recommended Next Improvements

  • Persist mock user role or clear mock tokens on app restart.
  • Add backend contract tests before disabling USE_MOCKS.
  • Add optional email/mobile validation if needed by the admin workflow.
  • Move login strings into localization for English and Urdu.
  • Consider adding a password visibility toggle if users request it.
