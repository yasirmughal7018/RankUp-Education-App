RankUp Education — Authentication & Login Logic

Web API, database model, JWT tokens, registration workflow, class hierarchy, and request flows

Prepared forRankUp Education workspace
ScopeWeb API (WebApi/) + Flutter Mobile (Mobile/)
Auth modelJWT access token + opaque refresh token (stored hashed in PostgreSQL)
Registration modelPending rows in app_users — no separate account_requests table

1. Executive Summary

  • All authenticated API calls use a Bearer JWT access token.
  • Login validates username + password against app_users. Only users with is_active = true and a non-null password_hash may sign in.
  • Registration creates an inactive app_users row. Admin approval activates the account, sets a password, and creates the role profile row.
  • Five roles are supported: SuperAdmin, SchoolAdmin, Teacher, Student, Parent.
  • The mobile app uses one shared login screen. Role is determined from the API response after authentication — not selected on the login form.
  • All API responses use the wrapper ApiResponse<T> with success, message, data, and errors.
Important: Extra registration form fields (email, school name, reason, etc.) are validated by the API but not persisted. Only username (mobile), display_name, role, requested_at, is_active, and password_hash are stored on app_users.

6. Registration Request Flow

Mobile LoginPage (account request form) → AuthController.requestAccountAccess() → ApiAuthRepository.requestAccountAccess() → POST /api/auth/register → AuthController.RegisterAsync() → AuthService.RegisterAccountAsync() → UserRepository.AddAsync(User.CreateRegistrationRequest(...)) → INSERT app_users (username=mobile, display_name, role, is_active=false, password_hash=NULL, requested_at=now())

7. Login Request Flow

Mobile LoginPage → AuthController.login(identifier, password) → ApiAuthRepository.login() → POST /api/auth/login { username, password } → AuthService.LoginAsync() 1. UserRepository.GetByUsernameAsync() 2. Attach profile context (student/teacher/parent/school) 3. user.EnsureCanLogin() — is_active + password_hash required 4. PasswordHasher.Verify() 5. IssueRefreshToken() → INSERT refresh_tokens (hashed) 6. JwtTokenService.CreateAccessToken() → ApiResponse<LoginResponse> → Mobile saves tokens (TokenStore) + user (AuthLocalDataSource)

8. Token Refresh & Logout

8.1 Refresh

Client POST /api/auth/token/refresh { refreshToken } → Hash token (SHA-256 hex) → Lookup refresh_tokens by token_hash → Verify not expired/revoked → Revoke old token → Issue new refresh + access JWT → Return AuthTokensResponse

8.2 Logout

Client POST /api/auth/logout { refreshToken } → Revoke matching refresh_tokens row → Mobile clears TokenStore + local user cache

8.3 Access token lifetime

Configured in appsettings.json → Jwt:AccessTokenMinutes (default 30 minutes). Refresh token lifetime is 30 days in AuthService.

12. Security & Rate Limiting

  • Passwords hashed with PBKDF2-SHA256 (100,000 iterations) via PasswordHasher.
  • Refresh tokens stored as SHA-256 hex only — raw token returned once to client.
  • Login endpoint rate-limited: LoginPermitLimit = 8 per 60 seconds.
  • Password reset always returns success message to prevent username enumeration.
  • JWT signing key must be ≥ 32 characters — configure in Jwt:SigningKey.
  • CORS policy RankUpEducation applied for configured origins.
