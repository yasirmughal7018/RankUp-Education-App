1. Overview & account states

RankUp supports these user-creation and access paths:

  1. Self-service registration — Student / Parent / Teacher submit → approval queue in app_user_approval → School/Campus may soft-approve → only Portal Admin activates → user sets password → Login.
  2. Directory create (Web) — Admin creates Student / Teacher / Parent / SchoolAdmin / CampusAdmin with no password → auto-approved → user sets password on first login.
  3. Add role to existing account — Directory create matching an existing mobile/CNIC adds a second role (multi-role) when combination rules allow.
  4. School / campus change (Web) — Teacher / Student / CampusAdmin request a change from Profile → confirmation → account locks → destination admin applies (PortalAdmin any; SchoolAdmin for their school; CampusAdmin for Teacher/Student into their campus) or any allowed reviewer rejects → account unlocks. Parent cannot request school/campus change (parents are not school-scoped).

PortalAdmin is not created via the app (seed/DB). SchoolAdmin is created by PortalAdmin only. CampusAdmin is created by PortalAdmin or SchoolAdmin.

Important: Approving a registration or directory-creating a user does not set a password. The user must set a password on first login, then sign in.

3. Ways to create users

Path A — Self-service → Approval queue → Portal Admin activates → Set password → Login

  1. Guest opens Request account access (Web or Mobile).
  2. Chooses Student / Parent / Teacher and submits.
  3. API creates pending app_users row + role in app_user_roles + one pending row per eligible reviewer in app_user_approval.
  4. Eligible admins see the request on Registration approvals.
  5. SchoolAdmin / CampusAdmin may Approve → their queue row is marked; account stays pending.
  6. PortalAdmin Approves → account activates (NeedsPasswordSetup); role profile created.
  7. User enters CNIC/mobile → sets password → signs in.

Path B — Directory create (Web only)

  1. Admin opens Directory (Students / Teachers / Parents / School Admins / Campus Admins).
  2. Creates user with username and required fields — no password.
  3. User is auto-approved (active, no password) → NeedsPasswordSetup.
  4. On first login, user sets initial password, then signs in.

Path C — Add role to existing user (multi-role)

  1. Admin creates Teacher / Parent / SchoolAdmin / CampusAdmin with a mobile or CNIC that already belongs to an existing account.
  2. If combination rules allow, the new role is added to app_user_roles (same login identity).
  3. User keeps one password; after login they can switch active role.

6. School / campus change (request, lock, unlock)

This is separate from registration. An already-active Teacher / Student / CampusAdmin can request a school and/or campus change from Profile. The request does not use the profile “Save” button — it has its own save action and confirmation. Parent cannot request a school/campus change (same rule as registration: parents are not assigned school/campus).

Request flow (Web)

  1. User opens /account → section School / campus change.
  2. Selects new school and/or campus → clicks Request school / campus change.
  3. Confirmation popup explains that the account will lock and they will be signed out.
  4. On confirm → POST /api/auth/me/school-change → pending request created → is_active=false → refresh tokens revoked → user logged out → redirected to /account-locked.

7. Directory (admin) create rules

No password on create. Directory users are auto-approved and must set password on first login (NeedsPasswordSetup). Directory create does not use app_user_approval.

Directory overview (Web)

  • Route: /admin/directory — summary tiles drive the active section (Schools, School Admins, Campus Admins, Parents, Teachers, Students).
  • Active list shows Active/Ready records only. Clicking a people tile opens a details popup (contact, audit fields, approval history when present). Header shows stats already visible on the tile; those fields are not repeated in the body.
  • Last login is shown on the right of active people tiles and in the popup header (date on one line, time on the next). Source: app_users.last_login_at, updated on successful password login (POST /api/auth/login).
  • School tile: one campus → school + campus details; multiple campuses → choose campus, then school + campus details.
  • School / campus changes button on Directory overview opens the pending school-change queue for PortalAdmin, SchoolAdmin, and CampusAdmin.

8. Multi-role & active role

One person can hold multiple roles on one login. Roles are stored in app_user_roles only.

Allowed combinations

  • Teacher + Parent
  • CampusAdmin + Teacher and/or Parent
  • SchoolAdmin + Teacher and/or Parent
  • SchoolAdmin + CampusAdmin + Teacher/Parent (allowed by rules)

Blocked combinations

  • Student cannot combine with any other role
  • PortalAdmin cannot combine with any other role

Session behaviour

  1. Login default active role = earliest assignment in app_user_roles.
  2. API returns role (active) and roles (all).
  3. POST /api/auth/switch-role issues new tokens for the selected role.
  4. Refresh token stores active_role so refresh keeps the same acting role.
  5. Authorization uses the active role only.

UI

  • Web: role switcher in the user menu when roles.length > 1.
  • Mobile: role switcher on Profile when multiple roles exist.

9. Login / password / active / locked checks

Shared flow (Web + Mobile)

  1. Enter CNIC or mobile → POST /api/auth/login-status.
  2. Branch: PendingApproval | Rejected | NeedsPasswordSetup | LockedPendingSchoolChange | Ready (or error “not active” for plain deactivation).
  3. For Ready only: Sign in → POST /api/auth/login.

Login identifier resolution: username → CNIC → mobile (prefers non-rejected row when a soft-rejected duplicate exists).

While SchoolAdmin / CampusAdmin have approved a registration but PortalAdmin has not, login-status remains PendingApproval.

While a school/campus change is pending and the account is locked, login-status is LockedPendingSchoolChange (not PendingApproval).

Web locked page

  • Route: /account-locked
  • Opened after submitting a school/campus change, or when login-status returns LockedPendingSchoolChange.
  • Shows why the account is locked and a link back to sign in.
Mobile: API already returns LockedPendingSchoolChange. Mobile UI should show the lock message (same as Web). If Mobile still treats unknown statuses like Ready, treat that as a defect until updated.

11. QA scenarios (step-by-step)

QA-01 Student self-request — Portal Admin activates

WebMobile
  1. Open Request account access.
  2. Account type = Student. Fill full name, mobile, roll; optionally school + campus.
  3. Submit request. Confirm app_user_approval rows match selection.
  4. As PortalAdmin, open Approvals → Approve.
  5. On login: enter mobile/CNIC → Needs password setup.
  6. Set password (≥6) → then Sign in.
Expected: Student reaches app home. Pending list cleared. Directory (Web) shows Active.

QA-02 Parent self-request — Portal Admin only

WebMobile
  1. Request as Parent; confirm school/campus/roll are hidden.
  2. Submit with name + mobile.
  3. SchoolAdmin / CampusAdmin Approvals → Parent request must NOT appear.
  4. PortalAdmin Approves → Parent sets password and logs in.
Expected: Only PortalAdmin sees and activates.

QA-03 Campus path — CampusAdmin records, PortalAdmin activates

WebMobile
  1. Request as Student or Teacher with school + campus.
  2. As CampusAdmin of that campus, Approve.
  3. Confirm account still PendingApproval (cannot set password yet).
  4. CampusAdmin list still shows the row; buttons hidden; message: Approved — awaiting Portal Admin.
  5. As PortalAdmin, Approve → NeedsPasswordSetup → set password → login.
Expected: CampusAdmin does not activate. PortalAdmin activates.

QA-04 School-only path — SchoolAdmin records, PortalAdmin activates

WebMobile
  1. Request as Teacher/Student with school selected and campus empty.
  2. CampusAdmin must NOT see it.
  3. SchoolAdmin Approves → still pending; awaiting Portal Admin.
  4. PortalAdmin Approves → activates.
Expected: SchoolAdmin alone does not activate registration; PortalAdmin does.

QA-05 PortalAdmin first — no need for School/Campus

WebMobile
  1. Submit Student/Teacher with school + campus.
  2. As PortalAdmin, Approve immediately (without Campus/School acting).
  3. Confirm NeedsPasswordSetup; user can set password.
Expected: PortalAdmin alone activates even if other approval rows remain pending.

QA-06 Teacher / Student without school (Portal only)

WebMobile
  1. Request as Teacher or Student with school empty (Student still needs roll).
  2. Confirm SchoolAdmin / CampusAdmin do not see it.
  3. PortalAdmin Approves → activates → set password.
Expected: Portal-only queue; activation works without school/campus on the request.

QA-07 Student request validation

WebMobile

  2. Without full name or mobile → blocked.
  3. Campus without school → blocked.
  4. School/campus empty is allowed (Portal-only queue).
Expected: Roll + identity required; school/campus optional for Student.

WebMobile

QA-10 Reject registration (soft-reject)

WebMobileAPI / DB

  2. Admin Rejects from Approvals.

Expected: Soft-reject only; trail kept; re-request → PendingApproval on new row.
Do not expect “No account found” after reject — that was the old hard-delete behaviour.
