1. Overview & account states

RankUp supports these user-creation paths:

  1. Self-service request — Student / Parent / Teacher submit → approval queue written to app_user_approval → admins record approvals → only Portal Admin activates → user sets password → Login.
  2. Directory create (Web) — Admin creates Student / Teacher / Parent / SchoolAdmin / CampusAdmin with no password → auto-approved → user sets password on first login.
  3. Add role to existing account — Directory create matching an existing mobile/CNIC adds a second role (multi-role) instead of creating a duplicate user (when combination rules allow).

PortalAdmin is not created via the app (seed/DB). SchoolAdmin is created by PortalAdmin. CampusAdmin is created by PortalAdmin or SchoolAdmin.

Important: Approving a registration or directory-creating a user does not set a password. The user must set a password on first login, then sign in.

3. Ways to create users

Path A — Self-service → Approval queue → Portal Admin activates → Set password → Login

  1. Guest opens Request account access (Web or Mobile).
  2. Chooses Student / Parent / Teacher and submits.
  3. API creates pending app_users row + role in app_user_roles + one pending row per eligible reviewer in app_user_approval.
  4. Eligible admins see the request on Registration approvals (“Pending with” lists remaining reviewers).
  5. SchoolAdmin / CampusAdmin may Approve → their queue row is marked; account stays pending.
  6. PortalAdmin Approves → account activates (NeedsPasswordSetup); role profile created.
  7. User enters CNIC/mobile → sets password → signs in.

Path B — Directory create (Web only)

  1. Admin opens Directory (Students / Teachers / Parents / School Admins / Campus Admins).
  2. Creates user with username and required fields — no password.
  3. User is auto-approved (active, no password) → NeedsPasswordSetup.
  4. On first login, user sets initial password, then signs in.

Path C — Add role to existing user (multi-role)

  1. Admin creates Teacher / Parent / SchoolAdmin / CampusAdmin with a mobile or CNIC that already belongs to an existing account.
  2. If combination rules allow, the new role is added to app_user_roles (same login identity).
  3. User keeps one password; after login they can switch active role.

5. Approval queue & activation rules

Who sees what

ReviewerSees in pending listCan act
PortalAdmin All pending requests Approve (activates) or Reject
SchoolAdmin Own school (school-only or with campus). Portal-only (no school) hidden. Approve (record only) or Reject — until they have already approved
CampusAdmin Own school + campus only. School-only and Portal-only hidden. Approve (record only) or Reject — until they have already approved

Activation hierarchy (critical)

Who approvesEffectAccount activated?
PortalAdmin Marks their app_user_approval row; creates role profile; sets is_active=true, must_change_password=true Yes — other pending rows are not required
SchoolAdmin Marks their queue row only No — still needs PortalAdmin
CampusAdmin Marks their queue row only; SchoolAdmin is then not required No — still needs PortalAdmin
  • After SchoolAdmin / CampusAdmin approve, the request stays on their Approvals list.
  • Approve / Reject buttons are hidden for that admin; UI shows: Approved — awaiting Portal Admin.
  • “Pending with” lists remaining pending reviewers from app_user_approval (approved_at IS NULL). If a CampusAdmin has already approved, pending SchoolAdmin rows are omitted from “Pending with”.
  • Student profile defaults on PortalAdmin activation: grade 1, section A.
  • On Reject: soft-reject — app_users.rejected_at set; rejector's app_user_approval row gets is_approved = false; other pending queue rows left as-is. Login-status returns Rejected; same identity may request access again (new user row).
  • Approvals UI (Web): school on line 1, campus on line 2; no Admin Target column; long reason truncated with Read more.

6. Directory (admin) create rules

No password on create. Directory users are auto-approved and must set password on first login (NeedsPasswordSetup). Directory create does not use app_user_approval.

Student Portal / School / Campus Admin fullName, username, school, campus, roll, grade, section mobile
Teacher Portal / School / Campus Admin fullName, username, school, campus, teacherCode mobile
Parent Portal / School / Campus Admin fullName, username cnic, mobile
SchoolAdmin PortalAdmin only fullName, username, school mobile, cnic, email
CampusAdmin PortalAdmin or SchoolAdmin fullName, username, school, campus mobile, cnic, email
  • Web: /admin/directory/school-admins, /admin/directory/campus-admins
  • CampusAdmin create/list is forced to own school for SchoolAdmin; CampusAdmin cannot manage Campus Admins.
  • If mobile/CNIC matches an existing user and role rules allow, the role is added to that account (Path C).

7. Multi-role & active role

One person can hold multiple roles on one login. Roles are stored in app_user_roles only.

Allowed combinations

  • Teacher + Parent
  • CampusAdmin + Teacher and/or Parent
  • SchoolAdmin + Teacher and/or Parent
  • SchoolAdmin + CampusAdmin + Teacher/Parent (allowed by rules)

Blocked combinations

  • Student cannot combine with any other role
  • PortalAdmin cannot combine with any other role

Session behaviour

  1. Login default active role = earliest assignment in app_user_roles.
  2. API returns role (active) and roles (all).
  3. POST /api/auth/switch-role issues new tokens for the selected role.
  4. Refresh token stores active_role so refresh keeps the same acting role.
  5. Authorization ([Authorize(Roles=...)] and service checks) use the active role only.

UI

  • Web: “Acting as” switcher in the header when roles.length > 1.
  • Mobile: role switcher on Profile when multiple roles exist.

8. Login / password / active checks

Shared 3-step flow (Web + Mobile)

  1. Enter CNIC or mobile → POST /api/auth/login-status.
  2. Branch: PendingApproval | NeedsPasswordSetup | Ready.
  3. Sign in → POST /api/auth/login.

Login identifier resolution: username → CNIC → mobile.

While SchoolAdmin / CampusAdmin have approved but PortalAdmin has not, login-status remains PendingApproval (account not active).

10. QA scenarios (step-by-step)

QA-01 Student self-request — Portal Admin activates

WebMobile
  1. Open Request account access.
  2. Account type = Student. Fill full name, mobile, roll; optionally school + campus.
  3. Submit request. Confirm app_user_approval rows match selection (Portal only / School+Portal / Campus+School+Portal).
  4. As PortalAdmin, open Approvals → Approve.
  5. On login: enter mobile/CNIC → Needs password setup.
  6. Set password (≥6) → then Sign in.
Expected: Student reaches app home. Pending list cleared for all admins. Directory (Web) shows Active.

QA-02 Parent self-request — Portal Admin only

WebMobile
  1. Request as Parent; confirm school/campus/roll are hidden.
  2. Submit with name + mobile.
  3. SchoolAdmin / CampusAdmin Approvals → Parent request must NOT appear.
  4. PortalAdmin Approvals → Approve (activates).
  5. Parent sets password and logs in on Web and Mobile.
Expected: Only PortalAdmin sees and activates. Parent becomes Ready after set-password + login.

QA-03 Campus path — CampusAdmin records, PortalAdmin activates

WebMobile
  1. Request as Student or Teacher with school + campus.
  2. As CampusAdmin of that campus, Approve.
  3. Confirm account still PendingApproval (cannot set password yet).
  4. CampusAdmin list still shows the row; buttons hidden; message: Approved — awaiting Portal Admin.
  5. “Pending with” shows Portal Admin (School Admin omitted after CampusAdmin approval).
  6. As PortalAdmin, Approve → NeedsPasswordSetup → set password → login.
Expected: CampusAdmin does not activate. PortalAdmin activates. SchoolAdmin not required after CampusAdmin.

QA-04 School-only path — SchoolAdmin records, PortalAdmin activates

WebMobile
  1. Request as Teacher/Student with school selected and campus empty.
  2. CampusAdmin must NOT see it.
  3. SchoolAdmin Approves → still pending; message Approved — awaiting Portal Admin.
  4. PortalAdmin Approves → activates.
Expected: SchoolAdmin alone does not activate; PortalAdmin does.

QA-05 PortalAdmin first — no need for School/Campus

WebMobile
  1. Submit Student/Teacher with school + campus (full queue).
  2. As PortalAdmin, Approve immediately (without Campus/School acting).
  3. Confirm NeedsPasswordSetup; user can set password.
Expected: PortalAdmin alone activates even if other approval rows remain pending.

QA-06 Teacher / Student without school (Portal only)

WebMobile
  1. Request as Teacher or Student with school empty (Student still needs roll).
  2. Confirm SchoolAdmin / CampusAdmin do not see it.
  3. PortalAdmin Approves → activates → set password.
Expected: Portal-only queue; activation works without school/campus on the request.

QA-07 Student request validation

WebMobile
  1. As Student, submit without roll number → blocked.
  2. Without full name or mobile → blocked.
  3. Campus without school → blocked.
  4. School/campus empty is allowed (Portal-only queue).
Expected: Roll + identity required; school/campus optional for Student.

QA-08 Parent must not send school/campus/roll

WebMobileAPI
  1. UI: switching to Parent clears/hides school fields.
  2. API optional: POST register as Parent with schoolId → validation error.
Expected: Parent requests never carry school/campus/roll.

QA-09 Duplicate mobile / CNIC / username

WebMobile
  1. Submit a valid Student request.
  2. Submit again with same mobile or CNIC → error.
  3. Reject first request, then same mobile can register again.
Expected: Uniqueness while pending/active; after reject, re-request allowed.

QA-10 Reject registration

WebMobile
  1. Create a pending request.
  2. Admin Rejects (before or without prior Campus/School record).
  3. Confirm removed from Approvals.
  4. Login with same identifier → status Rejected (message invites re-request).
  5. Submit Request Access again with same CNIC/mobile → succeeds (new pending row).
Expected: Soft-reject (user + approval trail kept); login-status Rejected; re-request allowed.

QA-11 SchoolAdmin & CampusAdmin scope isolation

WebMobile
  1. Create Student request for School A / Campus 1.
  2. Create Parent request (no school).
  3. SchoolAdmin of School B → must not see School A request.
  4. CampusAdmin of School A / Campus 2 → must not see Campus 1 request.
  5. CampusAdmin of Campus 1 → sees Student request; not Parent.
  6. SchoolAdmin of School A → sees Student request; not Parent.
Expected: Scope by school/campus on the request (not admin_target).

QA-12 PendingApproval login messaging (including partial approval)

WebMobile
  1. While request is pending (including after CampusAdmin/SchoolAdmin approval), enter identifier on login.
  2. Do not proceed to password login.
Expected: Status PendingApproval until PortalAdmin activates.

QA-13 NeedsPasswordSetup then Ready (after PortalAdmin)

WebMobile
  1. PortalAdmin Approves a pending user.
  2. Login identifier → NeedsPasswordSetup.
  3. Try login before set-initial-password → blocked.
  4. Set password → login succeeds → Ready.
  5. Later login goes straight to password step.
Expected: Set-password does not auto-login; user must sign in afterward.

QA-14 Directory create Student (auto-approve + set password)

WebMobile
  1. Admin opens /admin/directory/students → Create.
  2. Fill required fields including username — NO password field.
  3. Save; confirm first-login password message.
  4. Login: NeedsPasswordSetup → set password → Sign in.
Expected: No pending approval queue. NeedsPasswordSetup then Ready.

QA-15 Directory create Teacher / Parent

WebMobile
  1. Create Teacher via directory (no password); first login set password.
  2. Create Parent via directory (no password); optionally link student; first login set password.
Expected: Both Active after create; Ready after set-password + login.

QA-16 PortalAdmin creates SchoolAdmin

WebMobile
  1. PortalAdmin opens /admin/directory/school-admins.
  2. Create School Admin: name, username, school (optional contacts). No password.
  3. SchoolAdmin / CampusAdmin cannot open this page.
  4. New School Admin: NeedsPasswordSetup → set password → access school Approvals.
Expected: Auto-approved SchoolAdmin; first login requires set-initial-password.

QA-17 PortalAdmin / SchoolAdmin creates CampusAdmin

WebMobile
  1. PortalAdmin or SchoolAdmin opens /admin/directory/campus-admins.
  2. Create Campus Admin with school + campus (SchoolAdmin locked to own school). No password.
  3. CampusAdmin cannot open Campus Admins management.
  4. New Campus Admin: set password → Approvals only for own campus; can create Student/Teacher/Parent in own campus.
Expected: Auto-approved CampusAdmin; campus-scoped approvals and directory create.

QA-18 CampusAdmin creates Student / Teacher / Parent

Web
  1. Login as CampusAdmin (active role CampusAdmin).
  2. Create Student/Teacher — school/campus forced to own campus.
  3. Create Parent.
  4. Attempt create for another campus → blocked.
Expected: Creates succeed only in own school+campus; auto-approved; users need set-password.

QA-19 Multi-role: add Teacher to existing Parent

WebMobileAPI
  1. Create/approve a Parent who can login (Ready).
  2. As admin, directory-create Teacher using the same mobile (and school/campus/teacher code).
  3. Confirm no duplicate user; Parent account now has roles Parent + Teacher in app_user_roles.
  4. Login → default active role from earliest assignment; switch to Teacher on Web header / Mobile Profile.
  5. Confirm nav/permissions change with active role; refresh keeps active role.
Expected: One account, two roles; switch-role works; one password.

QA-20 Multi-role: CampusAdmin + Teacher

WebMobile
  1. Create CampusAdmin, then add Teacher role via directory (same mobile) OR create Teacher then add CampusAdmin.
  2. Login and switch between CampusAdmin and Teacher.
  3. As CampusAdmin: Approvals + directory available.
  4. As Teacher: quiz manage available; Approvals not available.
Expected: Active-role gates features correctly; both roles listed in /auth/me.

QA-21 Multi-role blocked: Student + anything / PortalAdmin + anything

WebAPI
  1. Try directory-create Teacher with mobile of an existing Student → error.
  2. Try add any role to PortalAdmin → error.
  3. Try add Student role to a Teacher → error.
Expected: Validation / business-rule errors; no second role added.

QA-22 Deactivate / activate user (active check)

WebMobile
  1. Create or PortalAdmin-approve a user who can login.
  2. On Web directory, Deactivate.
  3. Attempt login on Web and Mobile → not active.
  4. Activate again → login succeeds.
Expected: Deactivated is not PendingApproval. Activated returns to Ready.

QA-23 SchoolAdmin / CampusAdmin cannot create outside scope

WebAPI
  1. Login as SchoolAdmin; try create student for another school.
  2. Login as CampusAdmin; try create for another campus.
Expected: Blocked by UI and/or API (403 / validation).

QA-24 Approvals UI polish (Web)

Web
  1. Submit request with a long reason.
  2. Approvals table shows school line 1, campus line 2; truncated reason; Pending with list.
  3. Click Read more → popup shows full reason.
  4. Confirm Admin Target column is not shown.
  5. After CampusAdmin approve, confirm status text instead of buttons.
Expected: Truncation + popup; no Admin Target; awaiting message after local approve.

QA-25 Searchable school/campus on Request Access (Web)

Web
  1. Open /request-access as Student/Teacher.
  2. Open School dropdown → type to filter (optional field).
  3. Select school → Campus searchable list loads.
Expected: Filter-as-you-type works; Mobile may remain native dropdowns.

QA-26 Password minimum length

WebMobile
  1. After PortalAdmin approval, try set password shorter than 6 characters.
Expected: Validation error; password not saved.

QA-27 Login with CNIC vs mobile

WebMobile
  1. Register/create user with both CNIC and mobile when possible.
  2. After Ready, login using CNIC, then using mobile.
Expected: Both identifiers resolve to the same account when stored.

QA-28 Notifications for new registration (admin)

WebMobile
  1. As eligible admin logged in, submit a new request from another device.
  2. Check in-app notifications / Approvals (Portal / School / Campus as queued).
Expected: Eligible admins in the approval queue receive RegistrationRequest notification.

QA-29 Non-admin cannot open Approvals

WebMobile
  1. Login as Student / Teacher / Parent (active role).
  2. Navigate to Approvals / admin registrations.
Expected: Redirect / forbidden; no pending list.

QA-30 Switch role then refresh token

WebMobileAPI
  1. Login as multi-role user; switch to non-default role.
  2. Wait for / force access-token refresh (or call refresh API).
  3. Call GET /api/auth/me.
Expected: Active role remains the switched role after refresh; permissions match active role.
